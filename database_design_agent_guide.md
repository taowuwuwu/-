# 数据库设计工程指南 (Agent学习包)

## 核心公理 (必须遵守)
1. **查询驱动**: 表结构由高频查询场景决定，而非业务概念
2. **成本意识**: 强一致性=高延迟，根据场景选择一致性等级
3. **Schema即代码**: 表结构变更需走代码评审流程
4. **无银弹**: 每个设计决策都是权衡，记录取舍原因

## 一、数据分类与建模策略

### 1.1 四类数据模型
| 类型 | 特征 | 变更频率 | 设计策略 | 示例 |
|------|------|----------|----------|------|
| **配置类** | 基准参考数据 | 月/年级 | 单表+缓存，允许适度冗余 | 国家列表、币种、税率 |
| **状态类** | 实体当前状态 | 分钟/小时级 | 独立状态表，记录变更日志 | 订单状态、用户账户状态 |
| **流水类** | 不可变事件记录 | 实时高频 | 追加写，严禁更新删除 | 支付流水、操作日志 |
| **快照类** | 时间点数据副本 | 天/周级 | 定期生成，用于追溯分析 | 月末账单、日报快照 |

### 1.2 建表强制规范
```sql
-- 每个表必须包含的字段
CREATE TABLE example (
    id BIGINT PRIMARY KEY,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    is_deleted TINYINT NOT NULL DEFAULT 0,  -- 软删除标记
    version INT NOT NULL DEFAULT 1,         -- 乐观锁版本号
    data_level VARCHAR(20) NOT NULL         -- 数据分级: L1公开/L2内部/L3敏感/L4机密
);

-- 索引命名规范
-- idx_字段名_用途 (普通索引)
-- uk_字段名 (唯一索引)
-- idx_union_字段1_字段2 (联合索引)
```

## 二、存储形态决策树

### 2.1 写模型设计 (OLTP场景)
**原则**: 满足3NF，减少写入异常
- 每个非主键字段直接依赖主键
- 禁止传递依赖
- 多值属性必须拆分子表

### 2.2 读模型设计 (OLAP/查询场景)
**原则**: 反范式化，以空间换时间
- 高频JOIN字段冗余到主表
- 预计算聚合值 (如订单总金额)
- 使用JSON存储动态属性 (MySQL 5.7+)

### 2.3 冷热分离策略
| 触发条件 | 热数据 | 冷数据 | 迁移策略 |
|----------|--------|--------|----------|
| 时间维度 | 最近6个月 | 6个月前 | 定时任务归档 |
| 状态维度 | 进行中订单 | 已完成>1年 | 状态变更触发 |
| 访问频率 | 日访问>10次 | 日访问<1次 | 监控自动迁移 |

**冷数据访问层**: 应用层透明路由，业务代码无感知

## 三、一致性契约设计

### 3.1 一致性等级选择
| 场景 | 一致性要求 | 实现方案 | 性能影响 |
|------|------------|----------|----------|
| 资金交易 | 强一致 | 本地事务+分布式锁 | 高延迟 |
| 库存扣减 | 最终一致 | 消息队列+补偿 | 中延迟 |
| 用户画像 | 弱一致 | 异步批量更新 | 低延迟 |
| 社交点赞 | 可丢失 | 计数缓存+定期落库 | 极低延迟 |

### 3.2 幂等性设计模板
```java
// 所有写接口必须支持幂等
public void processOrder(OrderRequest request) {
    // 1. 检查幂等token
    if (!idempotentCheck(request.getToken())) {
        return; // 已处理过
    }
    
    // 2. 业务逻辑
    // 3. 更新幂等状态
}
```

### 3.3 降级预案清单
- 读降级: 返回缓存旧数据，标记"数据可能过期"
- 写降级: 写入消息队列，异步补偿
- 非核心功能熔断: 评论、推荐等功能可暂时不可用

## 四、五步落地流程

### Step1: 场景枚举 (输入)
列出所有查询场景及QPS预估:
```
- 查询用户订单列表: QPS=1000, P99<50ms
- 商家后台订单详情: QPS=200, P99<100ms  
- 财务对账报表: QPS=10, P99<5s
```

### Step2: 写模型构建
- 按3NF设计基础表结构
- 定义主键策略 (雪花ID/自增ID)
- 设置数据分级标签

### Step3: 读模型推导
- 针对高频查询添加冗余字段
- 设计覆盖索引避免回表
- 预计算聚合指标

### Step4: 异常注入测试
- 模拟主从延迟场景
- 测试并发写入冲突
- 验证慢查询告警

### Step5: 安全打标
- 敏感字段加密 (AES-256)
- 设置访问权限矩阵
- 开启审计日志

## 五、终极验收标准 (上线前必检)

## 六、常见陷阱与规避

| 陷阱 | 表现 | 后果 | 规避方案 |
|------|------|------|----------|
| 过度范式化 | 5张表JOIN查订单 | 查询性能差 | 冗余订单商品快照 |
| 大事务 | 单个事务更新10万行 | 锁等待超时 | 分批处理+消息队列 |
| 隐式转换 | 字符串字段查数字 | 索引失效 | 严格类型匹配 |
| 深分页 | LIMIT 100000,10 | 全表扫描 | 游标分页/ID范围 |
| 忽略字符集 | 混用utf8/utf8mb4 | 乱码/比较错误 | 统一utf8mb4 |

## 七、Agent决策辅助规则

### 规则1: 是否需要拆分表？
```
IF 单表数据量 > 500万 AND 日增量 > 1万
   AND 存在按时间范围查询需求
THEN 建议按时间分表或归档冷数据
```

### 规则2: 是否允许字段冗余？
```
IF 查询QPS > 100 
   AND JOIN表数量 >= 3 
   AND 冗余字段更新频率 < 查询频率/100
THEN 建议冗余并设计同步机制
```

### 规则3: 一致性等级选择
```
IF 涉及资金/库存/权益
THEN 必须强一致 (本地事务)
ELSE IF 影响用户体验但可修复  
THEN 最终一致 (消息队列)
ELSE 弱一致 (异步批量)
```

### 规则4: 索引创建判断
```
IF WHERE/ORDER BY/GROUP BY字段 
   AND 选择性 > 10% 
   AND 查询频率 > 100次/分钟
THEN 创建覆盖索引
```
